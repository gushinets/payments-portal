# ТЗ: первая RU-версия сайта для предъявления CloudPayments

## 1. Цель первой версии

Нужно быстро разработать первую RU-версию сайта платформы цифровых сервисов, чтобы показать ее представителям CloudPayments и подготовить основу для последующего подключения платежей.

Первая версия не является полноценным личным кабинетом и не должна включать полную продуктовую модель в БД. Основной фокус:

- показать понятный сайт с двумя продуктами;
- показать тарифы и сценарий оформления подписки;
- разместить обязательные юридические страницы;
- показать футер с реквизитами ИП и платежными методами;
- подготовить технические точки интеграции CloudPayments;
- сделать страницу результата оплаты;
- подготовить минимальный Docker/PostgreSQL-контур только для логирования webhook-событий.

  - `/ru/privacy`

## 4. Продукты первой версии

### 4.1. JobAct

Временное название: `JobAct`.

Тип продукта: mobile-first сервис / мобильный сценарий.

Описание:

> Цифровой акт выполненных работ: список работ, материалы, фото до/после, PDF на подпись, статус оплаты.

Ценность для пользователя:

- помогает быстро собрать акт выполненных работ;
- фиксирует состав работ и материалов;
- прикрепляет фото до и после;
- формирует PDF для подписи;
- помогает контролировать статус оплаты.

### 4.2. Scope Creep Guard

Временное название: `Scope Creep Guard`.

Тип продукта: Chrome Extension.

Описание:

> AI-сравнение нового запроса с исходным scope и генерация draft Change Order.

Ценность для пользователя:

- сравнивает новый запрос клиента с исходным объемом работ;
- подсвечивает потенциальный scope creep;
- помогает подготовить черновик change order;
- снижает риск неоплачиваемых дополнительных работ.

## 8. Требования к главной странице `/ru`

Главная страница должна сразу объяснять, что это платформа цифровых сервисов для оформления работ, контроля scope и подписочного доступа к инструментам.

Обязательные блоки:

- название платформы: AnytoolAI;
- краткое позиционирование;
- два продукта;
- тарифы;
- кнопки "Оформить" / "Попробовать";
- ссылка на юридические документы в футере;
- реквизиты ИП в футере;
- email поддержки.

Не делать сложный маркетинговый сайт. Первая версия должна быть простой, строгой и пригодной для проверки CloudPayments.

## 9. Требования к `/ru/auth-checkout`

Страница должна поддерживать product-aware entrypoint.

Примеры URL:

```text
/ru/auth-checkout?product=jobact
/ru/auth-checkout?product=scope-creep-guard
```

Если параметр `product` передан, страница сразу показывает выбранный продукт:

- название;
- описание;
- тариф;
- пробный период;
- кнопку оформления;
- email-форму;
- согласия;
- кнопку перехода к оплате.

Если параметр не передан, показать выбор из двух продуктов.

### Минимальный сценарий

1. Пользователь выбирает продукт или приходит по прямой ссылке с `product`.
2. Вводит email.
3. Отмечает чекбокс согласия на обработку персональных данных.
4. Нажимает "Получить ссылку для входа".
5. Система показывает экран "Мы отправили magic link на email".
6. Для первой версии допускается демонстрационный режим: кнопка "Продолжить как подтвержденный пользователь".
7. Пользователь видит тариф.
8. При необходимости включает автопродление отдельным чекбоксом.
9. Нажимает "Оплатить".
10. До подключения CloudPayments открывается `/ru/payment-result?status=demo`.

## 10. Magic link

В первой версии нужен magic link flow.

Минимально:

- форма ввода email;
- endpoint запроса magic link;
- экран "ссылка отправлена";
- endpoint подтверждения magic link;
- демонстрационный fallback, если отправка email еще не подключена.

Если полноценная email-отправка не готова, программист должен сделать dev/demo-режим:

- magic link отображается в логах backend;
- или на странице показывается кнопка "Демо: подтвердить email".

Для первой версии достаточно demo-режима без реальной отправки email.

## 15. CloudPayments: первая версия

Так как terminal id пока отсутствует, в первой версии нужно сделать оплату в режиме заглушки.

### Поведение кнопки оплаты до подключения CloudPayments

При нажатии на "Оплатить":

- не открывать реальную платежную форму;
- перенаправлять пользователя на `/ru/payment-result?status=demo`;
- показывать текст: "Оплата будет доступна после подключения CloudPayments".

### Поведение после получения terminal id

Программист должен предусмотреть конфигурацию:

```text
CLOUDPAYMENTS_PUBLIC_ID=
CLOUDPAYMENTS_API_SECRET=
CLOUDPAYMENTS_ENABLED=false
```

Если `CLOUDPAYMENTS_ENABLED=true` и `CLOUDPAYMENTS_PUBLIC_ID` заполнен:

- кнопка оплаты должна открывать виджет CloudPayments;
- сумма, валюта, описание и email должны передаваться в платежный виджет;
- назначение платежа должно быть понятным, например:
  - "Подписка JobAct Start на 1 месяц"
  - "Подписка Scope Guard Start на 1 месяц"

## 16. Webhook endpoints

Несмотря на отсутствие полной БД, нужно подготовить endpoint-структуру под CloudPayments.

Минимально:

```text
POST /api/cloudpayments/check
POST /api/cloudpayments/pay
POST /api/cloudpayments/fail
POST /api/cloudpayments/refund
POST /api/cloudpayments/recurrent
```

На первой версии endpoints должны:

- принимать запрос;
- сохранять payload в таблицу webhook events;
- возвращать корректный ответ-заглушку;
- не активировать реальную подписку;
- логировать ошибочные события.

Проверку подписи/HMAC заложить в коде, но если секрет еще не задан, работать в demo-mode.

## 17. Docker / PostgreSQL

Нужен `docker-compose.yml` с PostgreSQL.

Назначение PostgreSQL в первой версии: только хранение webhook-событий и технических логов платежного контура.

Пример минимальной таблицы:

```sql
create table payment_webhook_events (
  id bigserial primary key,
  provider text not null default 'cloudpayments',
  endpoint text not null,
  event_type text,
  invoice_id text,
  transaction_id text,
  account_id text,
  amount numeric(12, 2),
  currency text,
  raw_payload jsonb not null,
  headers jsonb,
  received_at timestamptz not null default now(),
  processed_at timestamptz,
  status text not null default 'received',
  error_message text
);
```

Полную модель пользователей, продуктов, тарифов, заказов и подписок в первой версии не делать.
