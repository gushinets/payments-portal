# MVP: путь пользователя, структура сайта и страницы платформы

## Обзор

Этот документ описывает MVP-вариант платформы с учетом двух дополнительных требований: вход по конкретному продукту и разделение на RU и Global версии сайта. Пользователь должен попадать не на нейтральную первую страницу, а на релевантный продуктовый сценарий, а региональный контур должен подбираться по geo/ip и языку браузера с возможностью ручного переключения.[cite:161][cite:164][cite:165][cite:174]

## Цель MVP

В MVP пользователь должен иметь возможность:

- перейти из конкретного extension сразу в релевантный сценарий;
- попасть в RU или Global контур;
- войти в систему;
- подтвердить email;
- ознакомиться с юридическими документами;
- выбрать продукт или тариф;
- оплатить подписку;
- получить доступ после подтверждения оплаты.[cite:91][cite:95][cite:100][cite:57]

## Структура сайта

Для MVP достаточно небольшого сервисного сайта с региональными версиями. Полный маркетинговый сайт на старте не обязателен, если есть понятный auth/checkout-контур, политика ПДн, оферта и продуктовые entrypoints.[cite:142][cite:144][cite:165]

### Минимальная структура

```text
/
├── /ru/auth-checkout        -> RU flow
├── /ru/privacy              -> RU политика обработки персональных данных
├── /ru/offer                -> RU оферта (включает условия автопродления)
├── /ru/cancellation         -> RU условия отмены подписки и возврата средств
├── /ru/cookies              -> RU политика куки
├── /ru/security             -> RU политика информационной безопасности
├── /ru/payment-result       -> RU возврат после оплаты
├── /en/auth-checkout        -> Global flow
├── /en/privacy              -> Global privacy policy
├── /en/offer                -> Global terms of service
├── /en/cancellation         -> Global cancellation & refund policy
├── /en/cookies              -> Global cookie policy
├── /en/security             -> Global security policy
├── /en/payment-result       -> Global payment result
└── /products                -> опциональный каталог продуктов
```

### Product-aware entrypoints

Рекомендуемый вход из extensions:

- `/ru/auth-checkout?product=ext_a`
- `/ru/auth-checkout?product=ext_b`
- `/ru/auth-checkout?bundle=scenario_1`
- `/en/auth-checkout?product=ext_a`
- `/en/auth-checkout?bundle=scenario_1`

Если пользователь приходит не из extension, а с общего сайта, можно использовать общий вход:

- `/ru/auth-checkout?entry=catalog`
- `/en/auth-checkout?entry=catalog`

## Geo-routing и выбор версии

Geo/IP и язык браузера можно использовать для первоначального определения рекомендуемой версии сайта, но лучше не делать это жестким редиректом без выбора. Пользователь должен иметь возможность вручную переключить регион.[cite:161][cite:164]

### Практическая логика

- Пользователь из РФ по умолчанию получает ссылку на RU-контур.[cite:161][cite:172]
- Пользователь вне РФ по умолчанию получает Global-контур.[cite:161]
- На первом экране есть переключатель `RU / Global`.
- Выбранный регион закрепляется в сессии.

## Доступность политик и документов

Политика обработки персональных данных должна быть размещена на отдельной странице сайта и быть доступной на каждой странице, где собираются персональные данные. Практический способ — держать ссылку на политику в футере сайта и дополнительно рядом с каждой формой ввода данных.[cite:150][cite:132][cite:156]

Под формами сбора данных должна быть видимая ссылка на согласие или на политику, а рядом с кнопкой отправки должен быть незаполненный чекбокс с текстом о согласии на обработку персональных данных.[cite:150][cite:154][cite:89]

### Футер (обязателен на каждой странице)

Футер должен содержать:

- Полное наименование юридического лица или ФИО ИП
- ИНН
- ОГРН (для ЮЛ) или ОГРНИП (для ИП)
- Юридический адрес
- Email службы поддержки
- Логотипы подключённых платёжных методов (только тех, которые реально подключены): МИР, T-Pay со ссылкой на tbank.ru, СБП
- Ссылки на все пять документов текущего регионального контура: `Политика ПДн` / `Privacy`, `Оферта` / `Terms`, `Условия отмены` / `Cancellation`, `Политика куки` / `Cookies`, `Политика ИБ` / `Security`

> CloudPayments при модерации проверяет совпадение данных в футере с данными в договоре. Домен должен быть зарегистрирован на то же юрлицо или ИП.

### Куки-баннер

При первом посещении сайта показывается куки-баннер:

- Информация о том, что сайт использует куки
- Кнопка «Принять»
- Кнопка «Настроить» (опционально, улучшает compliance)
- Ссылка на страницу `cookies`

Баннер показывается один раз; выбор пользователя сохраняется в localStorage или cookie.

### Как организовать доступность в MVP

- В футере всех страниц: юрданные, контакты, логотипы платёжных методов и ссылки на все пять документов.[cite:150][cite:132][cite:156]
- На странице `auth-checkout` в блоке логина: текст со ссылкой на `privacy`.[cite:150][cite:132]
- На шаге `consents`: явные ссылки на `privacy`, `offer` и `cancellation` перед кнопкой продолжения.[cite:132][cite:150]
- На шаге `checkout`: повторные ссылки на `privacy`, `offer` и `cancellation` рядом с кнопкой оплаты.[cite:150][cite:156]
- В product-aware first screen: ссылки на документы текущего регионального контура.
- Куки-баннер: отображается при первом посещении любой страницы сайта.

## MVP-путь пользователя

### Сценарий первого пользователя из extension

1. Пользователь в конкретном extension нажимает `Войти` или `Оформить подписку`.
2. Extension передает `product` или `bundle` в URL веб-контура.
3. Система определяет региональный контур по geo/ip и языку браузера, но дает возможность сменить регион.[cite:161][cite:164]
4. Пользователь открывает `auth-checkout` уже с информацией именно по тому продукту, из которого пришел.[cite:165][cite:174]
5. Пользователь вводит email.
6. Система отправляет код или magic link.
7. Пользователь подтверждает email.[cite:91][cite:95][cite:100]
8. Система показывает региональные документы и необходимые подтверждения / согласия.[cite:42][cite:45][cite:97]
9. После этого пользователь попадает на account/products state.
10. Пользователь видит продукт, по которому пришел, и при необходимости может раскрыть другие продукты.
11. Система показывает карточку выбранного тарифа и кнопку оплаты.
12. Пользователь переходит на hosted payment page соответствующего региону провайдера.[cite:58][cite:60][cite:64]
13. После оплаты пользователь возвращается на `payment-result` своего контура.
14. Backend получает webhook и активирует подписку.[cite:57][cite:58][cite:62]
15. Пользователь видит активный статус подписки и срок действия.

### Сценарий повторного пользователя

Если пользователь уже залогинен, email подтвержден, а актуальные документы уже приняты, система не должна снова прогонять его через весь сценарий. В этом случае пользователь сразу попадает на продуктовый экран или account/products state и может перейти к оплате или продлению.[cite:92][cite:94][cite:103]

## Подробное описание страниц MVP

### 1. `auth-checkout`

Это главная MVP-страница. В минимальной версии она закрывает почти весь сценарий: вход, подтверждение email, чтение документов, выбор продукта и запуск оплаты.[cite:80][cite:84]

#### Что должна делать страница

- Принимать пользователя из extension.
- Читать `product`, `bundle` или `entry` из URL.
- Определять региональный контур или принимать уже переданный route `/ru/` или `/en/`.[cite:161][cite:164]
- Показывать информацию прежде всего по тому продукту, из которого пришел пользователь.[cite:165][cite:174]
- Проверять наличие активной сессии.
- Показывать форму логина, если сессии нет.
- Запускать подтверждение email.
- Показывать региональные `privacy` и `offer`.
- Фиксировать принятие актуальных документов.[cite:42][cite:45][cite:97]
- Показывать список из 2 продуктов.
- Показывать статус подписки, если она уже есть.
- Давать возможность выбрать продукт и начать checkout.

#### Внутренние состояния страницы

| State | Что происходит |
|---|---|
| `product-intro` | Показ продукта, из которого пришел пользователь |
| `login` | Пользователь вводит email |
| `verify-email` | Пользователь подтверждает email |
| `consents` | Пользователь принимает документы / подтверждения |
| `account` | Пользователь видит аккаунт и продукты |
| `checkout` | Пользователь видит итог заказа и кнопку оплаты |
| `pending-payment` | Система ждет webhook |
| `active` | Подписка подтверждена и активна |

#### Что видит пользователь на первой странице

- Название продукта, из которого он пришел.
- Краткое описание продукта.
- Текущий тариф или доступные тарифы.
- Статус подписки по этому продукту.
- Кнопку `Оформить` / `Продлить`.
- Ссылку `Посмотреть все продукты`.
- Переключатель региона `RU / Global`.
- Ссылки на документы текущего контура.

### 2. `privacy`

Это обязательная публичная страница политики обработки персональных данных. Она должна быть доступна до отправки данных через login- или checkout-форму.[cite:43][cite:142][cite:144]

#### Что должна содержать

- Кто является оператором ПДн.
- Какие данные собираются.
- Для каких целей данные используются.
- Какие права есть у пользователя.
- Как связаться по вопросам ПДн.
- Региональные особенности соответствующего контура, если есть.

### 3. `offer`

Это обязательная публичная страница оферты. Она описывает правила предоставления цифрового сервиса и порядок оплаты подписки.[cite:137][cite:142]

#### Что должна содержать

- Что именно получает пользователь.
- Какие продукты и тарифы существуют.
- Как оплачивается доступ.
- Когда начинается и когда заканчивается подписка.
- Как работает продление или повторная оплата.
- Региональные условия соответствующего контура.

### 3a. `cancellation`

Это обязательная публичная страница условий отмены подписки и возврата средств. Для SaaS это не «возврат товара», а отдельный сценарий — CloudPayments проверяет наличие этой страницы отдельно от оферты.

#### Что должна содержать

- Как пользователь отменяет подписку (интерфейс, email, срок уведомления).
- Сроки прекращения доступа после отмены.
- Условия возврата денег: в каких случаях возможен, в какие сроки, каким способом.
- Что происходит с данными пользователя после отмены подписки.

### 3b. `cookies`

Это обязательная публичная страница политики использования куки.

#### Что должна содержать

- Какие куки используются: функциональные, аналитические, маркетинговые.
- Цель каждого типа куки.
- Как пользователь может отказаться от куки или настроить их.
- Список сторонних сервисов, устанавливающих куки (аналитика, платёжные системы).

### 3c. `security`

Это обязательная публичная страница политики информационной безопасности. CloudPayments требует её для подтверждения безопасной обработки данных карт.

#### Что должна содержать

- Общие меры защиты данных пользователей.
- Указание, что платёжный провайдер (CloudPayments) сертифицирован по PCI DSS.
- Явное указание, что данные карт вводятся на защищённой странице CloudPayments и не хранятся на серверах платформы.
- Использование HTTPS для всех соединений.
- Контакты для сообщений об инцидентах безопасности.

### 4. `payment-result`

Это обязательная техническая страница возврата после оплаты. Она нужна для UX, но не подтверждает сама по себе успешность платежа.[cite:57][cite:58][cite:60]

#### Что должна делать

- Показывать, что платеж обрабатывается.
- Сообщать, что доступ будет открыт после подтверждения оплаты.
- Опрашивать backend по статусу order или subscription.
- После подтверждения переводить пользователя в активное состояние.

#### Критичное правило

Подписка не должна активироваться только потому, что пользователь попал на `payment-result`. Источник истины — webhook платежного провайдера.[cite:57][cite:58][cite:62]

### 5. `products` (опционально)

Эта страница нужна только если каталог продуктов выносится из `auth-checkout`. Для MVP это не обязательно, особенно если продуктов всего два.[cite:108][cite:119]

#### Что должна делать

- Показывать оба продукта.
- Показывать короткое описание каждого.
- Показывать тариф или кнопку перехода в checkout flow.
- Показывать наличие активной подписки.

## Как фактически реализовать получение согласия на обработку ПДн

Для сайта нет одной жесткой «универсальной формы», но согласие должно быть конкретным, информированным и сознательным, а после изменений требований к согласиям нужно отдельно и явно оформлять нужные согласия, а не прятать их внутри других документов.[cite:51][cite:97][cite:153]

Под формой сбора данных должен быть незаполненный чекбокс, рядом с ним — текст о согласии и ссылка на политику или отдельную страницу согласия. Документы должны быть доступны до отправки формы.[cite:150][cite:132][cite:154][cite:89]

### Что должна сделать система технически

1. Показать пользователю product-aware first screen.
2. Показать региональный контур и дать возможность его переключить.[cite:161][cite:164]
3. Показать форму ввода email.
4. Под формой показать чекбокс без предустановленной галочки.[cite:156][cite:89]
5. Рядом с чекбоксом разместить текст согласия и ссылки на документы текущего контура.[cite:150][cite:132][cite:154]
6. На шаге `checkout`, если выбрана подписка с автопродлением, показать отдельный незаполненный чекбокс явного согласия на регулярные автосписания (требование Приложения 4 оферты CloudPayments).
7. Не отправлять форму, пока все обязательные чекбоксы не установлены.
8. При отправке зафиксировать:
   - user_id или временный session id;
   - региональный контур;
   - product / bundle entrypoint;
   - текст / тип документа;
   - версию документа;
   - timestamp;
   - ip;
   - user agent.[cite:56][cite:71]
8. Хранить лог принятия в отдельной таблице `document_acceptances`.

### Что должно быть в тексте согласия

Согласие должно содержать обязательные сведения: данные о субъекте, данные об операторе, цель обработки, перечень данных, перечень действий с данными, способы обработки, сведения о передаче третьим лицам при наличии, срок действия и порядок отзыва согласия.[cite:51][cite:152][cite:73][cite:158]


## Итоговая минимальная структура MVP

Минимальный MVP можно запускать со следующей структурой:

- RU и Global версии `auth-checkout`;
- RU и Global версии `privacy`;
- RU и Global версии `offer`;
- RU и Global версии `cancellation`;
- RU и Global версии `cookies`;
- RU и Global версии `security`;
- RU и Global версии `payment-result`;
- вход по `product` или `bundle` из extension;
- 2 продукта в каталоге;
- 1 payment provider на контур или соответствующий набор провайдеров;
- webhook-активация подписки;
- логирование принятия документов и согласий;
- футер с юрданными, контактами и логотипами платёжных методов на каждой странице;
- куки-баннер при первом посещении;
- HTTPS на всём домене;
- домен и хостинг зарегистрированы на то же юрлицо или ИП, что и договор с CloudPayments.[cite:43][cite:58][cite:64][cite:161]

Это минимальный набор страниц и механизмов, при котором пользовательский путь остается понятным, а MVP остается совместимым с требованиями по обработке ПДн, product-aware входом и разделением на RU/Global контуры.[cite:43][cite:125][cite:161][cite:172]
